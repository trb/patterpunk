# PatterPunk - a wrapper for large language models

I like the magentic library. The idea of just annotating functions
in python using the type annotations to grab output data felt really cool.

There's a few changes I wanted to make, though. The most important one was
to easily support chain-of-thought. For LLMs (large language models), the
context is their working memory, by getting them to "think out loud" as in
generate text that reads like internal thoughts, they're actually doing
something akin to "thinking".

But with the JSON generation forced into a function call, the LLM wasn't
able to generate a chain of thought. I tried to use output values like

```python
class WebResultChoice(pydantic.BaseModel):
    reasoning_for_choice: str
    url: str
    title: str
```

and sometimes it worked and `reasoning_for_choice` actually included some
chain of thought. But it was never as decent as a full non-function-call
response. I tried to hack JSON extraction from messages into magentic but
soon realized that the async support made things much more complicated for
my approach. Async is great for interactive LLM use, but I just need
a library to deal with LLMs programmatically, the whole async support only
got in my way.

So I decided to write my own library. I wanted to focus on programmatic use
of LLMs, ease of integration into existing codebases, and configurability
and extendability. Many libraries I've seen seem to only support
`temperature`, with some _maybe_ supporting `top_p`, and support for the
other parameters is rare (logit_bias can be very useful for classification).

## General approach

Another thing that many libraries seem to gloss over is that all functions
are always sent as an entire new context. It's totally possible to
switch models and parameters for messages in an ongoing chat. This is
helpful to e.g. have a cheaper model summarize text, a more capable one
extract meaning, and have the cheaper one wrap everything up in JSON, all
in one chat. Parameters and models should be per-message, not per-chat.

I also found that appending requests for function calls to more extensive
chats worked a lot better than just trying to get the LLM to generate
meaningful function calls directly.

PatterPunk supports two ways to generate JSON output: With function calls
and by parsing JSON inside of a message. By default, it'll scan the last
message in a chat for JSON, and try to deserialize it into the return type
annotation for your function. If it doesn't find JSON or the deserialization
fails it uses a cheap model to generate a function call. If that fails
it uses a more capable model to generate a function call. Only if that
fails as well does PatterPunk give up and return None.

## Name
I don't want to explain the name, I like that it's kinda weird and vague and that
there are multiple interpretations. I didn't come up with it, I picked it
from a list of names GPT-4 generated (figured that's appropriate for a
library dealing with LLMs..).

But here's an interpretation from GPT-4 that I condone:

> If we interpret "punk" in a more derogatory sense, and consider the library's function in relation to large language models (LLMs), the name "patterpunk" could suggest that the LLM (the "punk") is being used to generate "patter" - quick, fluent, possibly superficial or meaningless talk.
> 
> In this interpretation, the name could be a commentary on the nature of text generated by LLMs, perhaps suggesting that while the output is fluent and coherent (patter), it may lack depth, originality, or meaningful content, hence the LLM being referred to as a "punk". This interpretation would depend heavily on the creator's perspective on LLMs and their output.

There's also the "you little shit" interpretation of "punk", in the sense
that LLMs like to strictly follow requests in the literal sense, and end
up not doing what the user wants. And after particularly difficult
debugging and optimization session this can almost come off as malicious
